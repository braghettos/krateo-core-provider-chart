# core-provider — deployment overview (chart repo)

What core-provider is, and **how it deploys** as a Krateo composition. This is the deployment view;
the internals/runtime view (the crdgen pipeline, the composition-dynamic-controller reconcile loop)
lives in the code repo `braghettos/krateo-core-provider` (`docs/`). Every claim below is traced to a
file in this repo — if a comment disagrees with what the chart actually renders, the rendered chart
wins.

## What core-provider is

The Krateo **composition engine**. It watches `CompositionDefinition`s, reads each blueprint chart's
`values.schema.json`, **generates a typed CRD per blueprint** (via crdgen), and reconciles the
resulting Compositions — one composition-dynamic-controller (cdc) per generated CRD. It is the first
thing the installer brings up in bootstrap mode; every other Krateo component is a Composition driven
by it (`README.md:8-11`).

This repo is the **braghettos fork packaged as a Krateo blueprint**: the upstream Helm chart plus a
`values.schema.json` so `core-provider` can generate a typed CompositionDefinition CRD, and a
self-managing `CompositionDefinition` (`README.md:1-9`).

> **Requirement:** core-provider 2.0.x needs **Kubernetes ≥ 1.36** on the management cluster AND
> every remote target — it uses the GA `MutatingAdmissionPolicy` API (`admissionregistration.k8s.io/v1`)
> instead of an admission webhook, so there is no webhook server / serving cert
> (`README.md:30-34`, `chart/templates/compositions-version-policy.yaml:1-13`).

## Repo layout — four charts

| Path | Chart name | OCI artifact | Versioning |
|------|------------|--------------|------------|
| `chart/` | `krateo-core-provider` | `oci://ghcr.io/braghettos/krateo/krateo-core-provider` | tracks the git tag (`Chart.yaml` `version: CHART_VERSION`, `chart/Chart.yaml`) |
| `crds-subchart/` | `krateo-core-provider-crd` | `oci://ghcr.io/braghettos/krateo/krateo-core-provider-crd` | pinned literally (`crds-subchart/Chart.yaml`, currently `0.36.6`), independent of the app tag |
| `target-chart/` | `krateo-core-provider-target` | `oci://ghcr.io/braghettos/krateo/krateo-core-provider-target` | tracks the git tag (`target-chart/Chart.yaml` `version: CHART_VERSION`) |
| `kagent/chart/` | `krateo-core-provider-agent` | `oci://ghcr.io/braghettos/krateo/krateo-core-provider-agent` | `0.1.x`, independent (`kagent/chart/Chart.yaml`) |

The charts version **independently**:

- **The main app chart** (`chart/`) is the deployable core-provider engine + its sidecars. `version`
  is the `CHART_VERSION` placeholder substituted to the git tag at release; `appVersion` is the
  `APP_VERSION` placeholder, stamped from the code repo's latest semver tag, and is the default
  container image tag (`chart/Chart.yaml`, `chart/templates/deployment.yaml:36`
  `{{ .Values.image.tag | default .Chart.AppVersion }}`).
- **The CRD subchart** (`crds-subchart/`) holds the two **static** `core.krateo.io` CRDs and is
  deliberately **NOT bundled** into the app chart — per the golden rule, CRDs live in a dedicated
  chart deployed as its own Composition. The CRDs are generated by the **code repo's** CI
  (`controller-gen` via `go generate`) and pushed here by an automated PR on each release tag — do
  not hand-edit them (`README.md:21-28`). The subchart version is pinned literally
  (`crds-subchart/Chart.yaml`), NOT the placeholder, so the release workflow leaves it untouched.
- **The target chart** (`target-chart/`) ships only the composition-version `MutatingAdmissionPolicy`
  (`target-chart/templates/compositions-version-policy.yaml`); install it into every remote cluster
  referenced by a `KubernetesTarget` so compositions created there get stamped too (`README.md:30-37`).
- **The agent chart** (`kagent/chart/`) is the federated specialist agent (`krateo-core-provider-agent`)
  registered on `krateo-autopilot`; it versions on its own `0.1.x` line and is **not** the
  core-provider workload. `kagent/compositiondefinition.yaml` ships its CompositionDefinition (pinned
  `0.1.0`).

## The CompositionDefinition

Unlike component blueprints whose CompositionDefinition lives only in the installer umbrella,
core-provider ships a **self-managing** `CompositionDefinition` at the repo root
(`compositiondefinition.yaml`): `core.krateo.io/v1alpha1`, name `core-provider`, namespace
`krateo-system`, pointing `spec.chart.url` at `oci://ghcr.io/braghettos/krateo/core-provider`. This
lets core-provider manage its own upgrades as a composition once it is bootstrapped
(`README.md:39-54`). The deployed chart version is cluster-observable from
`CompositionDefinition.spec.chart.version` — this is the tag at which an agent should fetch THIS
repo's docs (see [llms.txt](llms.txt)).

The agent's own CompositionDefinition lives at `kagent/compositiondefinition.yaml`
(`core.krateo.io/v1alpha1`, name `krateo-core-provider-agent`, `spec.chart.version: "0.1.0"`).

## What the app chart deploys (`chart/templates/`)

Rendering the main `chart/` produces:

- **core-provider Deployment** (`deployment.yaml`) — one replica by default
  (`values.yaml:5`), the core-provider container on a single `http` port
  (`containerPort = service.port`, default 80, `deployment.yaml:43-45`, `values.yaml:42-44`). Config
  is delivered via `envFrom` the chart-managed env ConfigMap (`deployment.yaml:38-40`,
  `configmap.yaml:7-10`, rendering `.Values.env`, default `CORE_PROVIDER_DEBUG: "false"`). The
  Deployment also mounts four asset ConfigMaps read-only under `/tmp/assets/` — the cdc Deployment,
  cdc ConfigMap, cdc RBAC, and the JSON-schema ConfigMap (`deployment.yaml:52-71`); these are the
  **templates the engine stamps out for each per-composition cdc** at runtime.
- **chart-inspector sidecar** (`chart-inspector/deployment.yaml`, `service.yaml`,
  `serviceaccount.yaml`, RBAC) — a separate Deployment + ClusterIP Service on port 8081
  (`chart-inspector/service.yaml:11-13`). Image `ghcr.io/braghettos/krateo-chart-inspector`
  (`values.yaml:115-118`). core-provider calls it to inspect/expand blueprint charts.
- **cdc asset ConfigMaps** (`cdc-deployment.yaml`, `cdc-configmap.yaml`, `cdc-rbac.yaml`,
  `cdc-service.yaml`) — NOT a running cdc; they hold the templated Deployment / ConfigMap / RBAC /
  Service that core-provider renders **once per generated CRD** to spawn that composition's
  dynamic controller (`assets/cdc/*`). The cdc image is `krateo-composition-dynamic-controller`
  (`values.yaml:139-141`). `cdc-service.yaml` is only mounted when `cdc.metrics.enabled`
  (`deployment.yaml:62-66`).
- **Engine ClusterRole + binding** (`clusterrole.yaml`, `clusterrolebinding.yaml`,
  `serviceaccount.yaml`) — broad authority over `core.krateo.io` (compositiondefinitions,
  schemadefinitions, kubernetestargets), `customresourcedefinitions`, `deployments`, all RBAC
  objects, serviceaccounts, secrets (read), `composition.krateo.io/*`, configmaps, services, and
  events (`clusterrole.yaml:5-86`). The engine needs this because it creates the generated CRDs +
  per-composition cdc Deployments/RBAC.
- **Admission policies** (no webhook):
  - `compositions-version-policy.yaml` — a `MutatingAdmissionPolicy` (+ binding) stamping every
    `composition.krateo.io/*` CR on CREATE/UPDATE with the label
    `krateo.io/composition-version: <served version>`; consumed by core-provider for per-version
    listing/migration. CEL replacement for the former `/mutate` webhook — needs k8s ≥ 1.36
    (`compositions-version-policy.yaml:1-39`).
  - `compositions-max-name-length-policy.yaml` — a `ValidatingAdmissionPolicy` rejecting
    composition names > 44 chars (skipped for release-name-labelled objects,
    `compositions-max-name-length-policy.yaml:13-22`).
- **Optional** — HPA (`hpa.yaml`, `autoscaling.enabled: false`, `values.yaml:80-84`).

For the full `values.yaml` surface (resources, the nested `chartInspector.*` / `cdc.*` blocks,
exposure) and the operational gotchas, see [wiring.md](wiring.md). For the two static CRD fields, see
[crds.md](crds.md).
