# core-provider CRDs (chart repo)

The CRDs core-provider owns ship from the **`crds-subchart/`** chart (`krateo-core-provider-crd`,
`crds-subchart/Chart.yaml`), deployed as its own Composition independently of the app (see
[overview.md](overview.md)). This page is a **curated** field map — the authoritative, exhaustive
schema is the CRD YAML itself, and the Go source of truth is the code repo's `apis/`.

## What this repo ships

This chart owns **two static** CRDs:

| Kind | Group/Version | Scope | Plural | File |
|------|---------------|-------|--------|------|
| `CompositionDefinition` | `core.krateo.io/v1alpha1` | Namespaced | `compositiondefinitions` | `crds-subchart/templates/core.krateo.io_compositiondefinitions.yaml` |
| `KubernetesTarget` | `core.krateo.io/v1alpha1` | Cluster | `kubernetestargets` | `crds-subchart/templates/core.krateo.io_kubernetestargets.yaml` |

> **The per-blueprint Composition CRD is NOT here.** Each blueprint's typed CRD (Kind derived from
> the chart name, apiVersion from the chart version, e.g. `NginxChart` /
> `composition.krateo.io/v0-1-0`) is **generated dynamically by core-provider at runtime** via crdgen
> from the blueprint's `values.schema.json` — it is intentionally NOT shipped in this repo
> (`README.md:25-28`). If you need a generated CRD's shape, read the blueprint chart's
> `values.schema.json`, or `kubectl get crd` on the cluster.

Both CRDs carry `helm.sh/resource-policy: keep` so they survive chart uninstall and are generated by
`controller-gen` `v0.20.1` (`core.krateo.io_compositiondefinitions.yaml:5-7`,
`core.krateo.io_kubernetestargets.yaml:5-7`). CRD subchart version is pinned literally in
`crds-subchart/Chart.yaml` (currently `0.36.6`), independent of the app chart.

## CompositionDefinition — purpose

A `CompositionDefinition` is a **blueprint registration**: it points at a Helm chart; core-provider
reads that chart's `values.schema.json`, generates a typed CRD, and reconciles one Composition per
instance of that generated Kind (`compositiondefinitions.yaml:53`). Namespaced.

### `spec.chart` — the blueprint chart reference (the one required block)

`spec.chart` is required, and within it `url` is required
(`core.krateo.io_compositiondefinitions.yaml:74-128`).

| Field | Type | Purpose |
|-------|------|---------|
| `url` | string (required) | `oci` or `tgz` full URL of the blueprint chart. |
| `version` | string (≤20 chars) | Desired chart version; required for OCI charts and helm-repo URLs. |
| `repo` | string (≤256 chars) | Helm repo name (helm-repo URLs only; ignored for OCI/tgz). |
| `insecureSkipVerifyTLS` | bool | Skip TLS verification. |
| `credentials` | object `{username, passwordRef}` | Credentials for private repos; `passwordRef` is a `{key,name,namespace}` Secret reference (all three required). |

CEL `x-kubernetes-validations` on `spec.chart`: `version` is required once set, and `repo` is
required once set (immutability-style "required once set" rules,
`core.krateo.io_compositiondefinitions.yaml:124-131`).

### `spec.deploy` — local vs. remote target (core-provider 2.0.x)

`spec.deploy.targetRef.name` references a cluster-scoped **`KubernetesTarget`** to deploy the
generated CRD + cdc + RBAC to a remote cluster. **Omit `spec.deploy` for the default** — local
deployment in the management cluster (`core.krateo.io_compositiondefinitions.yaml:132-150`).

### `status` — what core-provider reports

Curated key status fields (`core.krateo.io_compositiondefinitions.yaml:151-294`):

| Field | Purpose |
|-------|---------|
| `apiVersion` / `kind` / `resource` | Last-applied identity of the generated custom resource. |
| `managed.group` / `managed.kind` | The generated CRD's Group/Kind. |
| `managed.versionInfo[]` | Per-chart-version info: the `chart` block, `version` (= the CRD version), `served`, `stored`. |
| `target.mode` | `Local` or `Remote` — where the controller is deployed. |
| `target.connectionStatus` | `Healthy` when the remote target is reachable, `Down` otherwise. |
| `target.version` | Kubernetes version reported by the target cluster. |
| `target.kubeconfigSecretResourceVersion` | resourceVersion of the kubeconfig Secret last used (credential-rotation traceability). |
| `digest` / `packageUrl` | Digest of managed resources; the resolved `.tgz`/oci chart URL. |
| `conditions[]` | Standard `Ready`/`Synced` conditions (also surfaced as printer columns). |

Printer columns expose `READY`, `SYNCED`, `AGE`, and (priority 10) `API VERSION`, `KIND`,
`PACKAGE URL`, `TARGET` (= `status.target.mode`), `CONNECTION`
(`core.krateo.io_compositiondefinitions.yaml:21-50`).

## KubernetesTarget — purpose

A **cluster-scoped** reference to a remote cluster, used by a CompositionDefinition's
`spec.deploy.targetRef` to deploy compositions remotely
(`core.krateo.io_kubernetestargets.yaml:24-26`).

| Field | Type | Purpose |
|-------|------|---------|
| `spec.kubeconfigRef` | object `{key, name, namespace}` (required) | Reference to a Secret key holding the **kubeconfig** of the target cluster. The Secret is the credential-rotation seam — populate/rotate it via External Secrets Operator (`core.krateo.io_kubernetestargets.yaml:48-67`). |

`spec.kubeconfigRef` is the only spec field, and it is required
(`core.krateo.io_kubernetestargets.yaml:68-70`).

## Exhaustive detail

- **The CRDs themselves:** `crds-subchart/templates/core.krateo.io_compositiondefinitions.yaml`
  (299 lines) and `core.krateo.io_kubernetestargets.yaml` (73 lines) — the authoritative schemas
  with every constraint and CEL rule.
- **The Go source of truth:** the code repo's `apis/` — the Go types from which these CRDs are
  generated by `controller-gen`. The dynamic crdgen pipeline (blueprint `values.schema.json` → typed
  Composition CRD) is documented in the code repo's
  [`docs/llms.txt`](https://github.com/braghettos/krateo-core-provider/blob/main/docs/llms.txt).
